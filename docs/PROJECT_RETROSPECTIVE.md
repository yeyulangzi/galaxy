## 一、项目概要

### 做了什么

从零搭建一个**个人知识图谱系统**，核心功能包括：

- 知识图谱可视化（D3-force + Canvas 2D）
- AI 内容投喂 → 自动抽取知识节点 → 人工审核入图
- 深度对话（Deep Dive）、全局聊天、多 Agent 人格
- 主动扫描（孤岛/缺口/老化检测）+ 反馈循环自进化
- 多维切面、思考版本、操作日志、安全模式
- 5 种 LLM Provider 适配（OpenAI / Anthropic / 阿里云百炼 / 火山引擎 / DeepSeek）

### 技术栈

Next.js 14 + React 18 + Tailwind CSS + Radix UI + Zustand + D3-force + SQLite + Drizzle ORM + pnpm Monorepo

### 最终规模

| 维度 | 数量 |
|------|------|
| 数据库表 | 15 张 |
| API 端点 | 40+ |
| 前端组件 | 28 个 |
| 代码行数 | ~35,000 行 |

## 三、产品视角复盘

### 3.1 AI 做产品的推荐流程

```
需求构思 → 设计文档（AI 辅助） → 里程碑拆分 → 逐里程碑实现 → 人工验收 → 迭代修复
```

**具体步骤**：

1. **需求描述**：用自然语言向 AI 描述你想要什么，不需要技术细节
2. **设计文档生成**：AI 生成设计 Spec（架构、数据模型、API 清单、UI 交互）
3. **里程碑拆分**：将大需求拆成 3~5 个可独立交付的里程碑
4. **逐步实现**：每个里程碑内，按 Schema → API → UI 的顺序推进
5. **人工验收**：每完成一个功能点，立即在浏览器中验证
6. **迭代修复**：发现问题后截图/描述给 AI，持续修复直到满意

### 3.2 里程碑拆分的经验

| 原则 | 说明 | Galaxy 中的体现 |
|------|------|----------------|
| **可独立运行** | 每个里程碑结束后都能跑起来 | M1 结束就有可用的图谱 |
| **渐进增强** | 后面的里程碑不破坏前面的 | M2 加 AI 不影响手动 CRUD |
| **人先机后** | 先做人工操作，再加 AI 自动化 | M1 手动 → M2 AI 投喂 |
| **先核心后周边** | 核心链路优先 | 先做投喂→审核→入图，再做扫描/反馈 |

### 3.3 产品经理在 AI 开发中的角色

| 传统角色 | AI 开发中的角色 | 变化 |
|---------|----------------|------|
| 写 PRD | 用自然语言描述需求 | 更轻量，但要更精确 |
| 画原型 | 口头描述 + AI 生成 + 截图反馈 | 不需要 Figma |
| 排期 | 按里程碑拆分，不做时间估算 | AI 速度不可预测 |
| 验收 | 实时在浏览器验收每个功能 | 更频繁，更即时 |
| 项目管理 | 管理 AI 的上下文和注意力 | **新职责** |

## 四、开发视角复盘

### 4.1 AI 生成代码的质量分布

| 质量等级 | 占比 | 说明 |
|---------|------|------|
| ✅ 首次可用 | ~40% | 简单 CRUD、Schema 定义、工具函数 |
| ⚠️ 需小修 | ~35% | 导入缺失、字段名拼错、类型不匹配 |
| ❌ 需大改 | ~15% | 逻辑错误、架构不合理、性能问题 |
| 🔥 完全重写 | ~10% | UI 设计不满意、方案选错 |

### 4.2 最常见的 Bug 类型

基于本项目 16 次 fix commit 的统计分析：

| Bug 类型 | 频次 | 典型案例 | 根因 |
|---------|------|---------|------|
| **字段名/变量名错误** | 5 次 | `edges.source_id` 应为 `edges.source_node_id` | AI 记忆 Schema 不精确 |
| **导入缺失/路径错误** | 3 次 | ESM/CJS 兼容、`.js` 后缀问题 | 不同模块系统的隐式规则 |
| **运行时环境差异** | 3 次 | webpack externalize、better-sqlite3 | AI 不了解具体运行环境 |
| **类型不匹配** | 2 次 | API 返回格式与前端期望不一致 | 跨文件类型推断弱 |
| **逻辑/数学错误** | 2 次 | 边线宽公式在低 weight 时接近 0 | AI 不做边界值分析 |
| **JSON 解析** | 1 次 | 推理模型返回非标准 JSON | AI 对第三方 API 行为假设过强 |

### 4.3 AI 擅长 vs 不擅长

| AI 擅长 ✅ | AI 不擅长 ❌ |
|-----------|-------------|
| 脚手架搭建（Monorepo、Schema、API） | UI 审美（需要反复迭代 3~4 轮） |
| CRUD 代码生成 | 跨文件一致性（字段名容易记错） |
| Provider 适配（模式代码重复度高） | 边界值处理（公式、数值映射） |
| 文档生成（README、交接文档） | 调试复杂运行时问题（webpack、ESM/CJS） |
| 重构和模式统一 | 理解用户的真实意图（需要多轮澄清） |
| 批量代码修改 | 删除/精简代码（倾向于加代码而非减） |

### 4.4 开发效率对比

| 维度 | 传统开发（1人） | AI 辅助开发 | 提效倍数 |
|------|----------------|------------|---------|
| Monorepo + Schema + API | 3~5 天 | 1 天 | **3~5x** |
| AI 抽取管线 + 多 Provider | 5~7 天 | 1 天 | **5~7x** |
| 图谱可视化 | 3~5 天 | 0.5 天 | **6~10x** |
| UI 设计迭代 | 2~3 天 | 1 天 | **2~3x** |
| Bug 修复 | — | 接近传统速度 | **1x** |
| **总计** | 15~25 天 | 4 天 | **4~6x** |

> **关键洞察**：AI 在「从 0 到 1 生成」阶段提效最大（5~10x），在「修复和调优」阶段提效最小（1~2x）。越到后期，越接近传统开发速度。

## 五、项目管理视角复盘

### 5.1 上下文管理是核心挑战

| 挑战 | 说明 | 应对 |
|------|------|------|
| **上下文窗口有限** | AI 无法同时记住 35,000 行代码 | 每次对话聚焦具体模块 |
| **跨会话记忆丢失** | 换一个 session AI 就忘了之前的决策 | 维护 HANDOVER.md 交接文档 |
| **字段名/类型漂移** | AI 在新代码中用错旧代码的名称 | Schema 和类型定义放在 shared 包，让 AI 随时查阅 |
| **代码库膨胀后变慢** | 代码越多，AI 理解越慢，错误越多 | Monorepo 隔离（db/ai/shared/web 各自独立） |

### 5.2 推荐的项目管理实践

#### ✅ 做的好的

1. **Monorepo 分包隔离**：让 AI 在一个包内工作时不需要理解其他包的全部细节
2. **Schema 集中管理**：`packages/db/src/schema/` 是唯一的数据源，AI 可以随时查阅
3. **交接文档 (HANDOVER.md)**：当 AI 上下文丢失时，文档就是恢复记忆的起点
4. **里程碑驱动开发**：每个里程碑有明确的交付物，避免范围蔓延
5. **截图反馈**：遇到 UI 问题直接截图给 AI，比文字描述高效 10 倍

#### ❌ 可以改进的

1. **缺少自动化测试**：只有 2 个 API 测试文件，大量功能靠人工验证
2. **UI 返工太多**：做了 3 次大改版（dark space → impeccable → clay），应该先确定设计系统再开发
3. **大文件问题**：`node-detail-panel.tsx`（1000+ 行）、`settings/page.tsx`（1200+ 行）早期应该拆分
4. **commit 粒度不均**：第一天 56 个细碎 commit，第二天 7 个巨大 commit

### 5.3 AI 辅助开发的成本模型

| 成本项 | 说明 |
|--------|------|
| **AI API 调用** | 主要成本，按 token 计费，复杂项目一天约 $5~15 |
| **人的时间** | 每天 6~10 小时（主要花在验收、描述需求、调试） |
| **返工成本** | UI 改版 3 次 ≈ 浪费 ~30% 的 UI 开发时间 |
| **技术债** | 大文件、缺少测试、snake_case 命名 — 需要后续还债 |

## 六、常见问题及解决方案

### Q1：AI 生成的代码跑不起来怎么办？

**现象**：AI 生成一大段代码，粘贴后报错。

**解决方案**：
- 不要一次性让 AI 生成太多代码。分步生成：先 Schema → 再 API → 最后 UI
- 每一步生成后立即运行验证，有问题立刻反馈
- 给 AI 提供错误信息（完整的报错堆栈），而不是只说"报错了"

### Q2：AI 记不住之前的设计决策怎么办？

**现象**：新 session 中 AI 使用了错误的字段名、API 格式等。

**解决方案**：
- 维护一份 **HANDOVER.md**，包含 Schema 定义、API 清单、关键设计决策
- 每次新 session 开始时，让 AI 先阅读 HANDOVER.md
- 关键字段名/类型集中定义在 shared 包中，AI 可以通过搜索找到

### Q3：UI 不满意，改了又改怎么办？

**现象**：AI 生成的 UI 要么太丑，要么风格不统一，导致多次返工。

**解决方案**：
- **先定设计系统**（颜色、字体、间距、圆角），再让 AI 按照系统生成组件
- 给 AI 一个参考截图或 URL（"像 Notion 的那种感觉"）
- 本项目的教训：做了 3 次 UI 大改版，应该在 M1 就确定 Clay Design System

### Q4：代码越来越多，AI 越来越慢怎么办？

**现象**：项目后期，AI 需要读取大量文件，响应变慢，错误增多。

**解决方案**：
- **Monorepo 分包**：让每个包保持在 AI 能理解的规模内
- **聚焦对话**：每次只讨论一个模块，不要混杂多个功能
- **文件拆分**：超过 300 行的文件考虑拆分

### Q5：如何判断一个功能该用 AI 做还是手写？

**判断标准**：

| 适合 AI 做 | 适合手写 |
|-----------|---------|
| 有明确模式的代码（CRUD、表单、列表） | 复杂的算法和数学公式 |
| 重复度高的代码（Provider 适配） | 精确的像素级 UI 调整 |
| 文档、注释、README | 性能关键路径的优化 |
| 脚手架和配置文件 | 涉及安全的加密/认证逻辑 |

### Q6：什么时候应该停下来不用 AI？

- AI 连续 3 次修不好同一个 bug → 自己看代码
- UI 返工超过 2 次 → 先画清楚再让 AI 实现
- 涉及金钱/安全/隐私 → 人工 review 每一行

## 七、给产品经理的 10 条建议

1. **先想清楚再开始** — AI 的速度会诱惑你"边做边想"，但方向错了越快越远
2. **用里程碑拆需求** — 不要一次性描述完整产品，给 AI 一个可交付的小目标
3. **先做核心链路** — 投喂 → 抽取 → 审核 → 入图，先跑通再加花
4. **截图比文字有效** — 遇到 UI 问题，截图圈出来，AI 秒懂
5. **维护一份交接文档** — 这是跨 session 的"记忆"，价值连城
6. **先定设计系统** — 颜色、字体、间距定好，后面就不用反复改
7. **频繁验收** — 每完成一个功能立即在浏览器确认，别攒到最后
8. **不要过度依赖 AI 的判断** — AI 会自信地写出错误代码，你需要批判性思维
9. **接受"够用就好"** — AI 生成的代码不完美，但 80 分能用就先往下走
10. **记录你的 prompt** — 好的需求描述可以复用，建立你的 prompt 库

## 九、结语

Galaxy 项目证明了一件事：**一个人 + AI，4 天可以做出一个中等复杂度的完整产品。**

但这不是"AI 自动写代码"的故事——而是一个**人与 AI 协作**的故事。人负责想清楚要做什么、验收质量、修正方向；AI 负责快速生成代码、处理重复工作、生成文档。

最大的效率提升不在于 AI 写代码有多快，而在于**它让你敢于尝试**。以前觉得"太复杂了不做了"的功能，现在可以说"让 AI 先写个版本看看"。这种心理门槛的降低，才是 AI 辅助开发最深远的影响。
